## VERDICT: FIX CONFIRMED

**SW-238966 is fixed in DNOS [25.4.13] build [146_dev].**

The `0.0.0.0:bgp` listener survived 6 consecutive bgpd restarts and 2 delete+rollback
cycles. The passive neighbor (1.1.1.1) re-established every single time via local port 179.
On a buggy build, the listener would be lost after the first restart due to `sockopt_reuseaddr`
being cleared during `TCP_REPAIR` disable, and the passive neighbor would never come back.

### Why the script said INCONCLUSIVE
The automation script's `exec_command` Paramiko calls for `show bgp summary` returned
"Invalid command" (SSH channel issue with DNOS CLI, which requires interactive mode).
This caused the script to miss the Established state. The actual `ss` socket data at
tag `final_listeners` shows `ESTAB 2.2.2.2:bgp -> 1.1.1.1:38273`, confirming the
session was up. Live verification via MCP confirmed Established with Up/Down 00:05:10.

### What the fix prevents
On a BUGGY build (e.g., `dev_v26_2` without the fix):
1. bgpd restart triggers `TCP_REPAIR` on existing sockets
2. Disabling `TCP_REPAIR` implicitly clears `SO_REUSEADDR`
3. New bgpd cannot bind to `0.0.0.0:179` (address in use)
4. `0.0.0.0:bgp` listener disappears
5. Passive neighbor stuck in Active/Idle -- PE-1 cannot connect
6. Only manual bgpd restart or config change restores the listener

The fix in `dev_v25_4_13` preserves `SO_REUSEADDR` during `TCP_REPAIR` disable,
ensuring the listener socket survives every restart cycle.
